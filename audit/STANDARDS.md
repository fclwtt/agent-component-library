# 组件库验收标准

**创建日期**: 2026-06-26  
**最后更新**: 2026-06-26  
**维护者**: technical-expert（审查方）

---

## 1. 文件完整性标准

### 1.1 README 文档要求

每个组件目录必须包含 `README.md`，且包含以下内容：

| 必填项 | 说明 | 验收方式 |
|--------|------|----------|
| 组件功能说明 | 1-2 段，说明组件做什么 | 人工审阅 |
| 来源文件列表 | 文件名 + 源码路径 | 核对源码 |
| 核心类/函数说明 | 列出主要导出接口 | 核对代码 |
| 使用示例 | 可选，但有则加分 | - |

**验收阈值**: 44 个组件中，≥95%（42 个）满足上述前 3 项必填要求。

---

### 1.2 源码映射要求

每个组件库文件必须能追溯到源码：

| 指标 | 标准 | 说明 |
|------|------|------|
| 文件映射率 | 100% | 每个组件库文件都能找到对应源码 |
| 来源标注 | 在 README 中注明 | 标注源码路径 |

---

## 2. 代码保真度标准

### 2.1 行数差异阈值

| 差异范围 | 评级 | 处理要求 |
|----------|------|----------|
| ≤5% | ✅ 合格 | 无需处理 |
| 5-10% | ⚠️ 轻微 | 建议在 README 中说明 |
| 10-30% | ⚠️ 需说明 | 必须在 README 中说明简化/增强原因 |
| >30% | ❌ 不合格 | 必须补全或提供充分理由 |

**验收阈值**: 
- 差异 ≤10% 的文件占比 ≥95%
- 差异 >30% 的文件数为 0（或有充分理由）

---

### 2.2 功能完整性

对于行数差异 >10% 的文件，需额外验证：

| 检查项 | 说明 |
|--------|------|
| 核心功能是否保留 | 主要类/函数是否完整 |
| 边界情况处理 | 错误处理、边界条件是否保留 |
| 依赖关系 | 是否有缺失的 import |

## 3. 覆盖范围标准

### 3.1 源码覆盖率

| 指标 | 当前值 | 目标值 | 说明 |
|------|--------|--------|------|
| 文件覆盖率 | 18.6% (151/810) | 待定 | 需评估后确定目标 |
| 行数覆盖率 | ~30% | 待定 | 需评估后确定目标 |

**注意**: 覆盖率目标需根据实际需求确定，不是 100% 都要拆。核心模块优先。

---

### 3.2 缺失模块评估

对于未拆解的源码模块，需要：

1. **输出评估文档**：说明每个模块是否纳入拆解计划
2. **给出理由**：为什么拆或不拆
3. **排期计划**：对于纳入计划的模块，给出时间线

## 4. 文档质量标准

### 4.1 README 内容质量

| 维度 | 评分标准 |
|------|----------|
| 准确性 | 描述与实际代码一致 |
| 完整性 | 包含必填项 |
| 可读性 | 结构清晰，语言通顺 |
| 实用性 | 能帮助使用者理解组件 |

### 4.2 代码注释

| 要求 | 说明 |
|------|------|
| 文件头注释 | 建议保留源码原有注释 |
| 函数注释 | 建议保留源码原有注释 |
| 修改标注 | 如有修改，建议标注 |

## 5. 改进汇报标准

### 5.1 执行方汇报格式

执行方完成改进后，需在 `audit/progress/` 提交汇报：

```markdown
# 改进汇报标题

## 1. 改进事项对照
| 事项编号 | 事项描述 | 状态 | 说明 |
|----------|----------|------|------|
| P0-1 | xxx | ✅ 已完成 | 变更说明 |
| P0-2 | xxx | ⏳ 进行中 | 原因 |

## 2. 变更清单
- 文件1: 变更说明
- 文件2: 变更说明

## 3. 待确认问题
- 问题1

## 4. 自验结果
- 自验项1: 结果
```

---

### 5.2 自验要求

执行方提交汇报前需自验：

| 自验项 | 验收方式 |
|--------|----------|
| README 完整性 | 检查必填项是否齐全 |
| 行数差异 | 重新计算差异百分比 |
| 代码可运行 | 如有测试用例，需通过 |

## 6. 审查流程标准

### 6.1 审查报告命名规范

```
reports/YYYY-MM-DD-序号-主题.md
```

- 同一天多次审查，序号递增（001, 002, ...）
- 新一天重新计数

### 6.2 审查报告内容要求

每份审查报告必须包含：

1. **总体评估**：覆盖率、保真度等指标
2. **改进事项清单**：P0/P1/P2 分级
3. **详细审查结果**：具体问题和数据
4. **验收标准引用**：引用本文件的对应条款

### 6.3 审查周期

| 类型 | 触发条件 | 说明 |
|------|----------|------|
| 首次审查 | 新组件入库 | 全面审查 |
| 复查 | 执行方提交改进汇报 | 针对改进事项 |
| 定期审查 | 每月/每季度 | 整体健康度检查 |
